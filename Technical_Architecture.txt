TECHNICAL ARCHITECTURE DOCUMENT
Tech Stack, Database Schema, Hardware & API Design

3. Hardware Components
3.1 RFID Attendance
Component
Role
RFID card
One per student/staff member, UID linked to their record in the students table
RFID reader + ESP32
Reads the card UID, sends it to the backend over Wi-Fi/Ethernet
Local storage on device
Queues scans if the internet is down, syncs automatically once reconnected — this is what rfid_scans.synced_at captures
3.2 GPS Bus Tracking
Component
Role
Driver's own phone
No dedicated hardware — runs the driver-facing view of the mobile app
Location permission + background tracking
Requested when the driver taps “Start Trip”; must handle Android/iOS background location rules correctly
Socket.io connection
Streams position updates to the backend, which pushes them to every parent/admin watching that route
4. API & Access Control Architecture
Every API request carries a JWT containing the user's role and school_id (see the earlier authentication discussion). Two layers of enforcement sit in front of every route:
	•	School isolation: every query is automatically scoped to the caller's school_id — no cross-school data can ever be returned, even by mistake.
	•	Role/scope check: middleware verifies the role (Principal/Teacher/Clerk/etc.) and, for teachers, whether the requested class is their in-charge class (full access) or a subject-only assignment (restricted to teacher_subject_assignments).
Representative endpoint groups

POST   /auth/verify-otp
POST   /auth/switch-account

GET    /students/:id

POST   /rfid/scan              (device API key, not a user JWT)
WS     /gps/bus/:busId          (Socket.io channel, live position)
5. Print & PDF Generation
	•	Server-side PDF generation (e.g., a headless-Chrome-based renderer) for report cards, absentee lists, and fee receipts — not the browser's raw print dialog, so formatting stays consistent regardless of device.
	•	Every list/report screen calls the same underlying “generate PDF” endpoint for its data, so the print feature is one reusable service, not rebuilt per screen.
6. Non-Functional Requirements Checklist
	•	Real-time: attendance, fee updates, and fee-structure changes propagate within seconds (Socket.io), not on a polling delay.
	•	Security: bcrypt password hashing for staff, OTP-only for parents, short-lived JWTs, rate-limited login attempts, audit log for sensitive actions.
	•	Data isolation: every table scoped by school_id, enforced at the query layer, not just the UI.
	•	Backups: automated daily RDS snapshots, S3 versioning for uploaded files.
	•	Offline resilience: RFID devices queue and sync scans; the mobile app should gracefully handle brief connectivity loss without losing in-progress actions.
	•	Extensibility: permissions and feature flags stored as data, not hard-coded (see Product Spec Section 14).
